Getting started

The framework is created to provide a solid UI base for all PIL AEM websites. The framework is front-end driven, platform agnostic, responsive, component based and compliant with the latest Philips Brand Identity guidelines.

Target audience

This technical documentation is meant for anyone who's interested in PIL AEM UI component development, varying from Philips UI developers to external agencies.

See the Revision history to get an overview on the latest document updates.

Scope

This document covers the state of the framework implemented in the Philips PIL AEM foundation-base module.

Component based

All code is component based, providing re-usable building blocks that are in line with the latest Philips Brand Identity guide lines.

Code structure:

  • Global CSS
    • Atomic components
      • Aggregated components

Responsive

The framework is fully responsive by design.

Breakpoints

Available breakpoints:

  • XS (extra small): < 451px
  • S (small): 451px - 700px
  • M (medium): 701px - 960px
  • L (large): > 960px

##Responsive images

Images needs to be fluid and adaptive. For each break point an optimized resolution of the image needs to be delivered. This will increase performance, as lower resolution images can be delivered to for example mobile devices.

As of 2016, browser support for the picture element is wide enough to use it in production, no longer requiring a solution like Picturefill. For the few browsers that do not support the picture element, the fallback image in the img tag will be loaded.

Browser and device support

Responsive design techniques should be employed to ensure that we build a flexible experience that looks good at all reasonable viewport sizes, not just specific device dimensions. The experience should be tested in the following browsers and devices:

  • Mobile
    • iPhone 5+ (iOS 9+, Safari)
    • Samsung Galaxy S 4+ (Android 4.2.2+, Google Chrome)
  • Tablet
    • iPad Air+ (iOS 9+, Safari)
    • iPad mini (iOS 8+, Safari)
    • Nexus 7+ (Android 4.4.2+, Google Chrome)
  • Desktop
    • Internet Explorer 11; Windows 7 and higher
    • Microsoft Edge 13+ (Windows 10)
    • Firefox (latest stable); Windows 7 and higher
    • Chrome (latest stable); Windows 7 and higher
    • Safari (latest stable); All MacOS versions where supported

Internationalization

RTL support

The framework supports RTL designs out of the box, by providing RTL-specific LESS files for framework essentials, atomics and modules. Page authors indicate that a page should be in RTL mode (html dir="ltr"). This is then used as a selector to apply RTL-specific LESS/CSS.

HTML5

doctype

The framework uses HTML elements and CSS properties that requires the HTML5 doctype to be on the top of your HTML page.

<!DOCTYPE html>

Mobile first

To make sure that all components are rendered properly on all devices, and touch zooming is enabled, add the following tag in the head:

<meta name="viewport" content="width=device-width, initial-scale=1">

Grid system

Responsive

The grid system is based on the following breakpoints:

  • XS (extra small): < 451px
  • S (small): 451px - 700px
  • M (medium): 701px - 960px
  • L (large): > 960px

Relative colum widths

The design specification follows the following convention regarding number of columns per breakpoint: As we are using relative class names these numbers are not set in stone from a technical point of view. By using class names like p-one-whole, instead of p-cols12 it doesn’t matter what the maximum designed amount of columns for a breakpoint is, one-whole always will mean one-whole (or 100% width), whereas p-cols12 means 100% on a 12 column grid, but 50% on a 24 column grid.

Horizontal spacing

Horizontal spacing between elements is controlled via the grid system. Since the grid doesn't have outer gutters by default, there will be no space between the grid and the viewport. To make sure that there is always is enough space between an element and the viewport, regardless of breakpoint, use the class below on row based elements:

  • p-row-gutter

The class takes care of proper spacing per breakpoint.

Theming

Theming is supported to enable the requirement to use color combinations and gradients for a group of pages. Within the group of pages, all components are affected by theming. Components have a default and inverted state. The inverted state is required to retain contrast and legibility, when positioned on light or dark backgrounds.

Theming is implementend using LESS CSS pre-compiler, which uses:

  • Variables
  • Built-in methods to calculate colors.

A theme consists out of a super set of 15 LESS color variables (@color1 - @color15), which may be used when creating a new theme.

Coding guidelines

These specific coding guidelines are an addition to the global Philips Digital front-end coding guidelines.

Javascript

  • All global variables should be added to the window.philips namespace (or a sub namespace, such as window.philips.utilities).
  • Following the Single Responsibility Principle, JavaScript components should be broken up into separate files, much like classes in traditional OO language.
  • These files should be minified and concatenated as part of the AEM clientlibs approach.
  • Components should follow the either the constructor, module, or jQuery plugin pattern, depending on use case.
  • Components (and thus instances) should have a public initialization method, allowing for (re-) initialization on demand.

CSS

Most best CSS best practices are intended to minimize payload, improve rendering efficiency, and promote readability. One thing often missing is an organizational approach. We intend to follow the principles of scalable and modular architecture for CSS, or SMACSS. Some additional recommended practices are as follows:

  • Only use class selectors for styling (Do not use ID selectors!)
  • Use as much as possible class based styling (over tag selectors), separating markup (structure) from styling. This allows flexibility of reorganizing the HTML markup without affecting the styling. Always apply this technique on sensitive SEO markup elements, such as headers (h1-h6).
  • Use !important for state only (this includes overriding inline CSS values set by JavaScript)
  • Use icon fonts and sprite sheets for background images. Use data URI's sparingly; only for small images (<4KB). Take theming into account. Use dynamically generated content (e.g., :before and :after pseudo-selectors) for simple images and in place of non-semantic DOM elements, where possible
  • Create class-based mixins for CSS animations and states to promote code reuse.
  • Keep CSS selector nesting levels as low as possible. Maximum levels allowed: 5.
  • Use LESS 1.3.0 as CSS pre-compiler language, since it’s natively supported by AEM 5.6.

SEO / Accessibility

  • Use legacy (X)HTML document outline approach which requires a strict heading structure: only a single <h1> tag per page is allowed. Use <h1> tag for primary document header and use logically <h2> to <h6> for subsequent headers. HTML5 document outline structure is not recommended, as the user agent (browser/crawler/screenreader) support is very limited. AEM integration takes care about the strict header structure.

IE8-10 support

As of 2016, IE8, IE9 and IE10 support has been dropped from the framework and our testing standards.

Framework behavior

Javascript

Browser resizes causes breakpoint event changes to be fired. But breakpoint event listeners will always receive the enforced Large breakpoint mode. This will not impact your functionality.

CSS

All framework base layout CSS (i.e. grid and atomic components) have been adapted to always render breakpoint large for IE8.

Component specific code

CSS Create IE8 specific styles for your component. Since the component CSS is mostly contained in media queries in LESS format, it’s quite easy to output breakpoint relevant CSS additionally for IE8.

Name convention

Most of what we need to know about naming conventions is defined in the Philip Digital front-end coding guidelines. Here are some additional points to remember:

  • Javascript variables, object properties (including namespace properties) should be camelCase. Exception to this rule is that a Javascript Class name should always start with a Capital.
  • CSS class names, LESS variables and mixins should be lowercased. Use hyphens as delimiters.
  • Filenames should be lowercase and hyphen-delimited.
  • Use prefix (“p-”) for all class names and id’s in Javascript and CSS.
  • Variables and namespace properties representing modules should parallel the names of the containing files, adjusting for these differences in convention.
  • The directory structure should parallel the namespace hierarchy As an example, /philips/components/atomic/checkbox.js contains a module assigned to the following namespace property: window.philips.components.atomic.checkbox.

Documentation

Use JSDoc3 comment style for inline Javascript documentation. See http://usejsdoc.org.

  • Document private functions where needed

Theming

Theming is implemented using LESS variables. These variables are related to color values and are used throughout the framework by its components. See the hierarchy below:

  • Global
  • Module

Global

All theming variables of the global code (/site/, /atomics/ and /libs/ folders) are stored in a single global variables file, located in [codebase]/less/site/variables.less.

The variables are logically grouped together and ordered from global to specific (top - bottom).

Module

Theming variables used in module specific code (i.e. in aggregated components) should be stored at the top in the applicable component LESS code. Since variables are lazy loaded in LESS, overriding variables can always be done. The last variable override wins.

See an example below for aggregated component d02-filters.less:

/*
Local variables
---------------------------*/
@d02-filters-border: @border-color;
@d02-filters-border-inverted: @border-color-inverted;

/*
Defaults
---------------------------*/
.p-d02-filters {
      border-color: @d02-filters-border;
}

...

/*
Inverted
---------------------------*/
.p-d02-filters {
    &.p-inverted,
    .p-inverted & {
          border-color: @d02-filters-border-inverted;
    }
}

Using variable reference assignments @d02-filters-border: @border-color allows overrides at multiple levels:

  • At global level: @border-color value can be changed and get propagated to all the components that use the variable.
  • At module (project) level: @d02-filters-border value can be changed specifically to override only that component specific color.
Rules
  • Prevent maintenance effort styling components using CSS and icon fonts, over using images.
  • Component variables should be named: [component-name-prefix]-[variable-name], for example: @d02-filters-border.
  • Use logic variable names consistently, look at the names used in site/variables.less. The variable name contains logic that defines the property and state. Sample of names used in variables: bg, hover, inverted, color, border etc.
  • Don't forget to add inverted style support for the component theming specific styles, when a component uses local theming variables.
  • Avoid using hardcoded values, but refer to global variables that are available in the site/variables folder.

Component coding guidelines

Important practices

  • Make sure the component is mobile first. Note that there are edge cases, where a desktop-first approach might suit better. Consult your teach lead when having difficulties choosing the correct approach.
  • Use responsive CSS grids for breakpoint specific behavior over Javascript DOM manipulation.
  • Use atomic components. Before starting a component, first check which elements are already available by the framework to avoid any double work.
  • Make sure the initial view of a component is rendered through server-side HTML and CSS, without the need for Javascript. This is best for UI rendering, avoiding unnecessary repaints.
  • Always watch the CSS output closely, especially taking IE8 into account.
  • Components should not have outer spacing (margin/padding) defined on the main component HTML element. Component spacing should be handled during integration on template level, respecting the layout/content model.

Versioning

In case of minor changes or patches, and backwards compatibility can be guaranteed, no new version of the component needs to be created. All changes can be applied within the existing component source files.

When there is a need for major changes that do have impact, and cannot support backwards compatibility, a new version of a component needs to be created. Use the following version strings "v[integer] inside the source file naming. Version integer starts at 2 and is incremented for each version: 2,3,4,5 etc. See a sample below using version 2:

  • ab01v2-my-first-component/optional-image.png
  • ab01v2-my-first-component.js
    • philips.components.aggregated.ab01v2MyFirstComponent
  • ab01v2-my-first-component.less
    • .p-ab01v2-my-first-component

IE8

See below how to deal with breakpoint specific styles in conjunction with IE8 support.

/*
Defaults
---------------------------*/
.p-form {
    background: red;
}

/*
Min Width 451
---------------------------*/
.form-s() {
   .p-form {
        border: 1px;
   }
}

/*
Min Width 961
---------------------------*/
.form-l() {
    .p-form {
        border: 2px;
   }
}

@media @S {
    .form-s();
}
@media @L {
    .form-l();
}

.ie8 {
   // We only want large breakpoint styles on top of default styling
   .form-l();
}

Make sure to put all LESS mixin calls in the right order in the IE8 specific selector.

Do not use IE8 specific styles in breakpoint mixin’s Don’t use any IE8 specific styles in mixins, but use them instead in the .ie8 selector. This avoids duplicated rules output. See an example below.

// Bad example:
.test-l()  {
  .test {
    color: red;
    .ie8 & {
      color: black;
    }
  }
}

@media @L {
  .test-l();
}

.ie8 {
  .test-l();
}

Then in IE8 you might end up with following CSS rules:

.ie8 .test { color: red; }              // THIS WILL BE APPLIED BUT SHOULD NOT
.ie8 .ie8 .test { color: black; }       // THIS WILL NOT BE APPLIED BUT SHOULD

So unless you move IE8 specific rule to an IE8 wrapper, you get a double .ie8 prefix in a rule.

// Good example:
.test-l()  {
  .test {
    color: red;
  }
}

@media @L {
  .test-l();
}

.ie8 {
  .test-l();
  .test {
    color: black;
  }
}

Then in IE8 you might end up with following CSS rules:

.ie8 .test { color: red; }         // THIS WILL BE APPLIED
.ie8 .test { color: black; }       // THIS WILL BE APPLIED AND OVERRIDES PREVIOUS RULE
